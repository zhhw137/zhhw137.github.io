服务治理（Netflix Eureka->Nacos）

基础架构

  • 服务注册：Eureka提供的服务端（spring-cloud-starter-netflix-eureka-server + @EnableEurekaServer），提供服务注册于发现的功能
  • 服务提供：Eureka提供的客户端（spring-cloud-starter-eureka-client+@EnableEurekaClient/@EnableDiscoveryClient,前者是只能在祖注册中心为eureka的时候使用），或者其他技术平台且遵循Eureka通信机制的应用。
  • 服务消费：消费者应用可以从服务注册中心获取服务列表，从而使消费者可以知道去何处调用其所需要的服务

服务治理机制

  • 服务提供者
    • 服务注册：“服务提供者”,即各个客户端，在启动时根据（eureka.client.register-with-eureka=true/false）判断，将自己注册到Eureka Server上，同时携带上自身服务的一些元数据信息。Eureka Server接受到信息后，存在一个双层结构Map上，第一层key为服务名，第二次key为实例名（负载均衡就是用一个服务中的多实例情况）
    • 服务同步：Eureka Server之间相互注册，服务提供者也分别注册到了俩个不同的服务中心上，当服务提供者发送注册请求到一个服务注册中心时，它会将该请求转发给集群相连的其他注册中心
    • 服务续约：根据俩个参数（eureka.instance.lease-renewal-interval-in-seconds=30 30秒发送一次心跳,eureka-instance.lease-expiration-duration-in-seconds=90 90秒未收到过期）这俩个参数定时向Eureka Server发送心跳,防止Eureka Server将该服务实例从服务列表中排除
  • 服务消费者
    • 获取服务：服务消费者发送REST请求给服务注册中心，来获取上面的服务清单。（eureka.client.fetch-registry=true/false 保证获取服务打开；eureka.client.registry-fetch-interval-seconds=30 设置为30秒更新一次服务清单）
    • 服务调用：Eureka中有Region和Zone概念，一个Region中可以包含多个Zone，每个服务客户端需要被注册到一个Zone中，所以每个客户端对应一个Region和一个Zone。进行服务调用时，优先访问同处一个Zone中的服务提供方，访问不到则访问其他的Zone。
    • 服务下线：服务实例进行正常的关闭操作时，触发一个服务下线的REST请求给Eureka Server，告诉服务注册中心下线。
  • 服务注册中心
    • 失效剔除：从服务列表中将因为内存溢出、网络故障等无法提供服务而没有给服务下线请求的服务剔除。Eureka Server启动时会创建一个定时任务，默认每隔60秒将当前清单中超时没有续约的服务剔除出去。
    • 自我保护：Eureka Server在运行期间，会统计心跳失败的比例在15分钟之内是否低于85%（主要实际生产环境通常由于网络不稳定导致），如果出现低于的情况，Eureka Server会将当前的实例注册信息保护起来，让这些实例不会过期，尽可能保护这些注册信息，因此，也可能导致客户端拿到实际已经不存在的服务实例，所以客户端要有容错机制，比如请求重试、断路器等机制。（eureka.server.enable-self-preservation=true/false 决定开关）

客户端负载均衡（Ribbon->LoadBalancer）

  • 服务端负载均衡

    • 硬件：f5
    • 软件：nignx:线性轮询、按权重负载、按流量负载

  • 客户端负载均衡

    • 服务提供者需要启动多个服务实例并注册到一个注册中心或是多个相关联的服务注册中心

    • 服务消费者直接通过调用被@LoadBalanced注解修饰过的RestTenplate来实现面向服务的接口调用

    • 负载均衡策略

      • RandomRule：随机策略: rand.nextInt(serverCount)函数来获取一个随机数，并将该随机数作为索引值返回具体实例（while循环里，正常要保证每次能选出一个）

      • RoundRobinRule：线性轮询的方式：在while循环中增加一个count计数变量，每次选择count++，当超过10次，结束尝试，打印警告

      • RetryRule：自动重试：线性轮询的方式，但是设置了尝试结束时间为阈值

      • WeightedResponseTimeRule:根据实例的运行情况来计算权重，并根据权重来挑选实例，以达到更优的分配效果。

        • 定时任务计算服务实例权重，默认30秒执行一次

        • 计算规则：根据LoadBalanceStats中记录的每个实例的统计信息计算：假设A、B、C、D，他们的平均响应时间为10、40、80、100，所以总响应时间是10+40+80+100=230，则每个权重为实例A：230-10=220，实例B为220+（230-40）=410，实例C为410+（230-80）=560，实例D：560+（230-100）=690，以此获得的每个实例的权重区间为

          • 实例A：[0,220]
          • 实例B：(220,410]
          • 实例C：(410,560]
          • 实例D：(560,690)

          随机一个在[0,690)之间的数字，该数字落在哪个区间， 使用哪个实例

      • ClientConfigEnabledRoundRobinRule：默认使用RoundRobinRule策略，一般作为父类，供其他的策略扩展自己的策略，当不满足扩展的策略的触发条件时，使用父类默认的RoundRobinRule的线性轮询策略

      • BestAvailableRule：利用负载均衡统计对象LoadBalancerStats计算，当前并发最小的，即最空闲的实例

      • AvailabilityFilteringRule:线性轮询的方式取出实例，再判断实例：是否故障，即断路器是否生效已断开；实例的并发请求数大于阈值，默认2的32次方-1，可以通过..ActiveConnectionsLimit来修改。10次后采用父类的实现方案。

      • ZoneAvoidanceRule:

        • 采用主过滤条件对所有实例过滤并返回过滤后的实例清单
        • 依次使用次过滤条件列表中的过滤条件对主过滤条件的结果进行过滤
        • 每次过滤之后，判断符合下面任一条件，将结果返回供线性轮询算法选择
          • 过滤后的实例总数>=最小过滤实例数（默认1）
          • 过滤后的实例比例>最小过滤百分比（默认0）

    • 重试机制：eureka选择了AP，则为了补偿可用性，加入重试机制。spring.cloud.loadbalancer.retry.enabled=true/false 来打开

服务容错保护（Hystrix->resilience4j/sentienl）

  • 服务熔断（当下游的服务因为某种原因突然变得不可用或响应过慢，上游服务为了保证自己整体服务的可用性，不再继续调用目标服务，直接返回，快速释放资源。如果目标服务情况好转则恢复调用。需要说明的是熔断其实是一个框架级的处理，而基本上业内用的是断路器模式；）

  • 服务降级

    • 开关降级：当下游的服务因为某种原因响应过慢，下游服务主动停掉一些不太重要的业务，释放出服务器资源，增加响应速度
    • 限流降级：系统的调用达到阈值，则主动或者被动，即上游或者下游服务，主动降级。
    • 熔断降级：当下游的服务因为某种原因不可用，上游主动调用本地的一些降级逻辑，避免卡顿，迅速返回给用户

  • 创建请求命令（以下的所有功能都可以继承类HystrixCommand的形式实现）

    • 注解形式：同步执行和第二个方法是异步执行

  • 定义服务降级

    • 不需要降级逻辑的：执行写操作的命令，执行批处理或离线计算的命令

  • 异常处理

    • 异常传播：通过指定注解，将指定异常包装在HystrixBadRequestException中抛出，这样则不会触发后续fallback逻辑。没有指定，则除了HystrixBadRequestExcepiton之外的其他异常，均触发fallback逻辑
    • 异常获取:在fallback实现的方法的参数中增加Throwable e对象的定义，就可以在方法内部获取触发服务降级的具体异常内容了

  • 命令名称、分组以及线程池划分

  • 请求缓存

    • @CacheResult为请求命令开启缓存功能(在没有@CacheKey的情况下，默认用改方法所有的参数)，也可以通过指定cacheKeyMethod方法来指定。
    • @CacheKey 设置方法哪些参数作为缓存的key（优先级低于CacheMethod方法指定key，有该方法时失效）
    • 缓存清理：commandKey 属性必须配置，让Hystrix定位正确的请求命令缓存位置

  • 请求合并

    • 不适合进行请求合并的场景：请求命令本身延迟不高、延迟时间窗内的并发量不高

声明式服务调用（Feign->OpenFeign）

  • 注解@FeignClient定义feign客户端 启动类上加@EnableFeignClients启用feign客户端。
  • 自带Ribbon和Hystrix
  • 请求压缩功能和日志记录请求功能

API网关服务（Zuul->GateWay）

  • 请求路由

    • 主要用Ant风格 ？单个字符、*任意数量字符、**匹配任意数量字符，支持多级目录

    • 路由加载顺序是按照写的时候的顺序，用LinkedHashMap存储

    • ignored-patterns（待定）

    • 路由前缀、本地跳转、Cookie等属性传递

    • 自带Hystrix和Ribbon

  • 请求过滤

    • 请求生命周期（也代表的是四种过滤器）
      • pre：进行请求路由之前的一些前置加工
      • routing：将外部请求转发到具体服务实例上，当服务实例将请求结果都返回之后，阶段完成
      • post：获取到请求信息和服务实例赶回信息，可以对处理结果进行一些加工或转换
      • error：上述3个阶段发生异常触发，处理后通过post过滤器发送出去

  • 动态加载

    • Config作配置中心 作动态路由
    • Groovy构建 动态过滤器

分布式配置中心（Config->Nacos）

  • 基础架构
    • 远程Git仓库:用来存储配置文件的地方。
    • Config Server：分布式配置中心，上面去指定要连接的Git仓库位置以及账号、密码等连接信息（配置文件bootstrap.properties中配置的应用名{application}、环境名{profile}、分支名{label}）
    • 本地Git仓库：本地用于存储每次客户端请求获取的配置信息，当远程Git仓库无法访问时，用本地Git库提供配置信息
  • 扩展功能
    • health 健康监测
    • 属性覆盖：spring.cloud.config.server.overrides.（属性名），保证该属性不会被config server中读取的属性覆盖
    • spring security 安全保护
    • 加密解密：安装JCE使用，要加密的属性加上{cipher}例如：spring.datasource.password={cipher}xxx
    • 高可用：作为一个服务，纳入Eureka的服务治理体系中
    • 失败快速响应与重试：spring.cloud.config.failFast=true，可以保证失败后不需要多等待前置的一些加载时间，实现快速返回失败信息；再引入spring-retry和spring-boot-starter-aop依赖，失败后会自动根据配置参数进行重试，重试过了阈值，再返回失败信息。
      • spring.cloud.config.retry.multiplier:初始重试间隔时间 默认1000毫秒
      • spring.cloud.config.retry.initial-interval:下一间隔的乘数，默认1.1，即第二次尝试1100毫秒
      • spring.cloud.config.retry.max-interval:最大间隔时间,默认2000毫秒
      • spring.cloud.config.retry.max-attempts：最大重试次数默认为6次
    • 动态刷新配置：spring cloud bus去配置变更通知

消息总线（Bus->Nacos）

  • 原理：某一个实例的更新配置请求会发送消息给消息总线，消息总线再根据配置通知所有的相关应用更新配置
  • 指定刷新范围
  • 架构优化：不再针对具体实例，而是加入Config Server到消息总线中，然后通过Config Server触发配置更新的职责。

消息驱动的微服务（Stream）

  • 开启绑定功能：@EnableBinding(value={A.class,B.class})通过value指定的Class类型数组，指定多个消息通道的绑定

  • 绑定消息通道：@Input和@Output注解定义消息通道，默认使用方法名作为消息通道的名称，也可以使用value属性指定；输出通道的返回值得是MessageChannel，而输入通道的返回对象得是SubscribableChannel

  • 消费组与消息分区

    • 消费组：默认生产者发出的消息绑定到通道上时，会给每一个消费者实例生成一个副本供其消费。可以用spring.cloud.stream.bindings.input.group属性实现分组

    • 消费分区

分布式服务跟踪（Sleuth）

  • ELK
  • Zipkin 可以以http方式 也可以集成spring cloud stream中间件消费信息
  • Zipkin原理
    • Span 一个基础工作单元：以HTTP请求为例，一次完整的请求活成，客户端和服务端会产生多个不同的Annotation，对于同一个请求来说，算作一个Span
    • Trace 它是由一系列具有相同Trace Id的Span串联形成的一个树状结构。在复杂的分布式系统中，每一个外部请求通常会产生一个复杂的树状结构的Trace
    • Annotation：用来记录一个时间的存在，即一个包含有时间戳的事件标签。
      • cs（client send）：记录客户端发起了一个请求，标识了这个HTTP请求的开始
      • sr（Server Received）：记录服务端接收到了请求，并准备开始处理。它与sr之间的时间戳之差即为http请求的网络延迟
      • ss（Server send）：用来记录服务端处理完请求后准备发送请求响应信息。与sr计算，服务端处理请求的时间消耗
      • cr（Client server）：记录客户端接收到服务端的恢复，与cs计算，客户端发起到服务端响应的总时间消耗
