说明

《kafka权威指南》的读后总结

初始Kafka

  • 消息：kafka的数据单元（可以看成是数据库里的一个“数据行”或一条“记录”），消息由字节数组组成，其有一个可选的元数据键，键也是一个字节数组。当消息以一种可控的方式写入不同分区时，使用键生成一个一致性散列值，再对分区取模，为消息选取分区（正常通过键的分配时固定的，但是如果增加分区，则可能导致新的键分配到别的分区，即分区导致计算结果有变动）。
  • 批次：一组消息；一个批次发送可减少网络开销，但是单个消息传输时间变长
  • 模式：kafka的消息格式；为了让生产者和消费者解耦，而特别保证的数据格式一致性
  • 主题：kafka的所有消息都必须有一个主题topic，类似于数据库的表或者文件系统的文件夹。主题内部不保证消息顺序性
  • 分区：一个主题可以划分为若干个分区，一个分区就是一个提交日志。消息以追加的方式写入分区，保证顺序性。分区可以分布在不同的服务器上，kafka也是通过分区来实现的数据冗余和伸缩性。
  • 生产者：创建消息的用户（客户端）。生产者一般会将一个消息发布到一个特定主题上，而不关心其具体落在哪个分区。但是也可以通过消息键（上面的消息中的那个键字节数组）和分区器（简单理解就是算法，可自定义），将同类消息落到一个分区
  • 消费者：去读消息的用户（客户端），消费者订阅一个或多个主题，按顺序消费消息。
  • 偏移量：消息的偏移量是一种元数据，一个不断递增的整数值，每个分区唯一。消费者把每个分区最后读取的消息偏移量保存在zookeeper（默认端口2181）或者kafka（默认9092）中，保证消费者不重复消费
  • 消费者群组：一个或多个消费者组成一个群组，共同读取一个主题，群组保证每个分区只能有一个消费者使用，同时当一个消费者群组失效，由群组的其他消费者接管失效消费者的工作。
  • broker：一个kafka服务器称为broker。broker接收生产者产生的消息，为消息设置偏移量，并提交到磁盘保存。broker为消费者提供服务，对读取分区的请求作出响应，返回已经提交到磁盘上的消息
  • 集群：broker是集群的组成部分，每个集群中都有一个broker同时充当了集群控制器的角色，负责管理工作，包括将分区分配给broker和监控broker。
  • 保留消息：kafka重要特性，默认保留策略，保留7天，或者保留的消息达到一定大小的字节数。

  • 发送消息方式

    • 发送并忘记：把消息发送给服务器，但并不关系是否正常到达。（但kafka高可用，会自充尝试重发，但是仍可能丢失一些消息）

    • 同步发送：send()方法发送消息，它会返回一个future对象，调用get()方法进行等待，就可以知道消息是否发送成功。

    • 异步发送：调用send()方法，并指定一个回调函数，服务器在返回响应时调用该函数。

消费者

  • 消费者群组：由多个消费者组成,一个消费者群组里的消费者不可以消费用一个分区，不同消费者群组里可以消费同一个分区。消费者群组的主要目的是，便于扩展消费者，在生产者产出大于消费速度时，可以增加消费者的形式增加消费速度。

  • 再均衡：分区的所有权从一个消费者转移到另一个消费者

    • 检测机制：消费者通过向被指派为，群组协调器的broker（不同的群组可以有不同的协调器）发送心跳（轮询消息或是提交偏移量时发送心跳）来维持他们和群组的从属关系以及他们对分区的所有权关系。（正常时间间隔发送心跳，则被认为活跃）
    • 分配分区：第一个加入群组的消费者为群主，它从协调器那里获得群组的成员列表（列表中包含了所有最近发送过心跳的消费者，即活跃的），并给每一个消费者分配分区。分配完成后，协调器发送给每个消费者自己的分配信息，给群组所有消费者的分配信息。

  • 创建消费者（只列举了必须属性）

  • 订阅主题

  • 轮询：第一次调用poll()方法时，负责查找GroupCoordinator,然后加入群组，接受分配的分区。轮询期间：再均衡、心跳、获取数据。（为了保证上述功能都执行，最好是线程安全的，例如，把消费者的逻辑封装在自己的对象里，然后用JAVA的ExecutorService启动多个线程，使每个消费者运行在自己的线程上）

  • 其他属性

    1. fetch.min.bytes:消费者从服务器获取记录的最小字节数，小于该值则不返回，等攒够了返回，降低消费者来回取数次数，降低其工作负载
    2. fetch.max.wait.ms:消费者获取记录的时间限制，默认500ms，如果数据一直不够上面的最小字节数条件，则达到这个时间条件的也返回，不用等待最小字节数
    3. max.partition.fetch.bytes:服务器从每个分区里返回给消费者 的最大字节数，该值需要比broker能够接收到的最大消息的字节数大，防止消费者无法读取这些信息。同时要保证值不会太大，防止单次poll返回的数据太多，处理时间过长，无法及时进行下一个轮询避免会话过期。
    4. session.timeout.ms和heartbeat.interval.ms:消费者被认为可以与服务器断开连接的时间，默认3s。heartbeat.interval.ms即心跳频率一般被设置为session过期的三分之一。session过期时间小，频繁轮询可能导致均衡过多非预期，时间大，会导致检测节点崩溃需要更长时间
    5. auto.offset.reset：指定消费者在读取一个没有偏移量的分区或者偏移量无效的情况下该做的处理
      • 默认值：latest,从最新的记录开始读取
      • earliest：偏移量无效时，从起始位置读取分区的记录
    6. enable.auto.commit: 设置为true则是自动提交偏移量，配合auto.commit.interval.ms属性控制提交的频率。
    7. partition.assignment.strategy：群组内分区分配给消费者的策略
      • Range（默认） : 将主题的若干个连续的分区分配给消费者。如消费者C1、C2，主题T1、T2有3个分区，则C1分T1和T2的分区0,分区1，C2分T1和T2的分区2（因为分区数量无法被消费者数量整除，就会第一个消费者获得的分区数量大于第二个消费者）
      • RoundRobin：将主题的所有分区逐个分配给消费者。如消费者C1分到T1的0、2分区和T2的1分区，C2获得T2的0、2分区和T1的1分区。（一般都会分配相同数量分区或最多差一个）
    8. client.id：broker用来标识从哪个客户端发送过来的消息，可以是任意字符串，通常用在日志、度量指标和配额里。
    9. max.poll.records：用于控制单次调用call()方法能够返回的记录数量，可以帮你控制在轮询里需要处理的数据量
    10. receive.buffer.bytes和send.buffer.bytes:socket在消费者和生产者读写数据时用到的TCP缓冲区大小，设为-1即为操作系统的默认值

  • 提交和偏移量：消费者每次调用poll()方法时，都会更新分区当前位置（偏移量），叫作提交。

    • 具体实现为，消费者向叫做_consumer_offset的特殊主体发送消息，其包含每个分区的偏移量，消费者不掉线，则该偏移量其实没什么用。消费者崩溃，触发再均衡，为了能够继续之前的工作，消费者需要从此读取每个分区最后一次提交的偏移量，然后从该位置开始继续处理。
    • 如果提交的偏移量小于客户端处理的最后一个消息的偏移量，那么俩者之间的消息会被重复处理
    • 如果提交的偏移量大于客户端处理的最后一个消息的偏移量，那么俩者之间的消息会丢失

  • 自动提交

    • enable.auto.commit:true/false 是否开启自动提交偏移量
    • auto.commit.interval.ms 自动提交的时间间隔
    • 可以设置为false关闭自动提交，手动调用commitSync()方法，同步将poll()返回的最新偏移量提交，该方法会阻塞当前线程一直重试
    • 手动调用commitAsync()方法可以，异步将poll()返回的最新偏移量提交，但是其不会进行重试。取而代之的是有回调函数，可以查看具体错误原因（一直重试可能于最新提交的偏移量冲突，比如最新的3000，这次是2000重试了之后覆盖了3000）（可以用序列号之类的辅助手段判断是否有新线程提交，有则不重试，否则重试）
    • 同步和异步组合提交

  • 提交特定的偏移量：消费者API允许在调用commitSync() 方法和commitAsync()方法时传进去希望提交的分区和偏移量的map

  • 再均衡监听器（实现ConsumerRebalanceListener接口）（consumer.subscribe(topics,new HandleRebalance())）(第二个参数为监听器接口实现类，通过该方法注册监听器)

    • public void onPartitionsRevoked(Collection partitions) 在再均衡开始之间和消费者停止读取消息之后被调用。（这里一般要提交偏移量，保证下一个接管分区的消费者知道从哪里开始读取）
    • public void onPartitionsAssigned(Collection partitions) 在重新分配分区之后和消费者开始读取消息之前被调用

  • 特定偏移量处处理记录

    • seekToBeginning(Collection partitions) 和 seekToEnd(Collection partitions)这俩个方法可以跳到分区开始和末尾读取消息
    • seek()方法可以定位到正在使用的位置，为了保存位置和消费记录的原子性，加入数据库进行存储

  • 退出：通过另一个线程调用consumer.wakeup()，这个方法是消费者唯一一个可以从其他线程里安全调用的方法。

  • 反序列化器

    • 自定义（不推荐）
    • 使用Avro反序列化

  • 简单消费者（不使用群组和再均衡，自己给自己分配分区）

深入kafka

  • 集群成员关系：broker启动时，通过创建临时节点，把自己的ID注册到Zookeeper（zk的/brokers/ids路径），当有broker加入或退出集群则会获得通知。一个ID只会在ZK上存在一个，如果改broker被关闭，其对应的节点会消失，但是ID仍然存在，并在下一次某个broker用同样的ID注册时，使用该ID之前对应的主题和分区。
  • 控制器：额外具有负责分区首领选举功能的broker，有borker节点加入或者退出集群时通知控制器，控制器负责在节点加入或离开时进行分区首领选举。当控制器broker被关闭或者断开连接，其他broker通过watch对象获得消息，他们会尝试让自己成为控制器，第一个在zk中创建控制器节点的broker会成为控制器，其他节点收到节点已存在异常
  • 副本
    • 首领副本
    • 跟随者副本：不停的从首领副本处复制消息，保持一致状态，以便首领崩溃时成为新首领。
      • 顺序复制首领消息，请求消息N时，首领就默认消息N-1之前的都已经同步。
      • 超过10s没有任何请求消息，则默认为不同步状态，首领奔溃时，其不会成为首领
    • 首选首领：创建主题时选定的首领，就是分区的首选首领。（其实就是分区副本清单里的第一个副本）。一般会让首选首领即为首领副本（当其俩者是同步状态时）（因复制算法需要）
  • 处理请求
    • 元数据请求：该请求包含客户端感兴趣主题列表，向任意broker请求。获得响应消息，其包括这些主题具有的分区、每个分区都有哪些副本、以及哪个副本是首领。客户端通过保存这些消息，并不时发送改消息更新状态，保证生产请求和获取请求能发给首领副本
    • 生产请求：客户端要写入broker的消息，只能发给首领副本生效，其他副本则会返回一个“非首领错误”，则发送元数据请求更新，找到首领副本
      • acks配置参数，指定写入成功的界定
        • 1 首领收到消息则为写入成功
        • all 所有同步副本收到消息才算写入成功
        • 0 不需要broker的响应都算成功
    • 获取请求：同生产请求一样，需要发送给首领副本。
      • 需要获取的数据或偏移量不存在或已删除则返回一个错误
      • broker会按照客户端指定的分区大小上限和下限以及超时时间限制返回指定数量数据。使用零复制技术向客户端发送消息（即直接通过linux文件系统缓存发送到网络通道）
      • 读取的数据只能是所有同步副本已经复制的消息，否则一旦首领崩溃，下一个首领没有这些数据，则会破坏一致性
  • 文件管理：kafka管理员为每个主题配置了数据保留期限，规定数据被删除之前可以保留多长时间，或者清理数据之前可以保留的数据量大小。kafka将分区划分为若干个片段，每个片段默认包含1GB数据或者一周数据，以较小的为准。当前正在写入数据的片段称为活跃片段，其永远不会被删除。
  • 索引：kafka每个分区维护一个索引，索引把偏移量映射到片段文件和偏移量在文件里的位置。其也被分为片段，所以删除消息也会删除索引。如果索引出现损坏，则kafka重新读取消息并录制偏移量和位置生成索引。

可靠的数据传递

  • 高可用保证
    • 核心：复制
      1. 生产者
        • 同步副本和非同步副本的划分（一旦副本变为不同步状态，自动重启非同步副本试图保持同步）
        • 消息必须等到所有同步副本都接收消息才会认为是已提交
        • 复制系数设置3及以上，保证不会因为1的时候的重启导致主题不可用。2的时候因为重启导致集群不不稳定，而重启导致集群不可用。
        • 不完全的首领选举（开启还是禁用）：首领宕机时，其他副本不可用或者不同步；则要么不允许首领选举成功，等待旧首领恢复（较低可用性），要么将新首领当成首领（导致数据丢失和出现数据不一致）
        • 最少同步副本，通过设置参数 min.insync.replicas，确定只有当集群中满足最少可用副本数时，才可以接受生产者消息，否则为只读状态。
        • 生产者重试机制（不过要又额外的类似版本限定，防止产生重复消息）
      2. 消费者
        • 可靠性配置：group.id:唯一的group.id保证消费者可以看到主题的所有消息；auto.offset.reset保证在没偏移量的时候从消息最开始或者最后面开始；enable.auto.commit和auto.commit.interval.ms:让消费者基于任务调度自动提交偏移量（但是要注意自动提交可能存在重复处理消息，比如在自动提交前就停止处理消息了；或者没处理完毕就提交偏移量漏掉消息）；
  • 不被重复消费(仅仅一次)
    • 把结果写到一个支持唯一键的系统里，比如键值存储引擎、关系型数据库、elasticsearch等
  • 可靠性传输
    • 通过kafka提供的org.apache.kafka.tools包里面的VerifiableProducer和VerifiableConsumer这俩个类，来模拟为生产者和消费者
    • 应用程序模拟测试：客户端从服务器断开连接、首领选举、依次重启broker、依次重启消费者、依次重启生产者
    • 生产环境监控
      • JMX度量指标：其中的error-rate和retry-rate如果上升说明系统出现问题
      • 监控生产者日志：如果生产者的消息剩余的重试次数为0，则说明生产者已经没有多余的重试机会，则要么增加次数，要么解决问题。
      • 消费者监控：consumer-lag：该指标表明了消费者的处理速度与最近提交到分区的偏移量之间的差距，理想状态为0
      • 监控数据流：给消息增加时间戳（0.10.0版本之后都有），应用程序记录生成消息的数量，消费者记录已读取消息的数量以及消息生成事件到当前时间之间的时间差。再使用工具比较生产者和消费者记录的消息数量，确保没有消息丢失，同时时间差也在我们允许范围内。（为了更好的监控，可以通过增加监控消费者，订阅一个特别的主题，只进行消息的计数操作来准确监控）
  • 顺序性：偏移量实现的
